01 — Definición

¿Qué es el Ciclo
de Vida del Software?

"El Ciclo de Vida del Software (SDLC) es un proceso estructurado que describe las etapas por las que pasa un sistema de software, desde su concepción inicial hasta su retiro definitivo, garantizando calidad, eficiencia y control en cada fase."

También conocido como SDLC (Software Development Life Cycle), este marco de trabajo proporciona un enfoque sistemático para planificar, crear, probar y entregar software de alta calidad, minimizando riesgos y costos.

🎯
Propósito
Organizar el desarrollo de software de manera estructurada y predecible, reduciendo fallos y asegurando la entrega de valor.
📐
Marco Metodológico
Define roles, responsabilidades, actividades y artefactos para cada etapa del proceso de desarrollo.
🔄
Carácter Iterativo
Aunque lineal en algunos modelos, el ciclo implica retroalimentación constante para adaptarse a cambios y nuevos requerimientos.
📊
Control de Calidad
Integra verificaciones y validaciones en cada etapa para garantizar que el producto final cumpla los estándares requeridos.
02 — Elementos

Elementos Clave

01
👥
Personas
Stakeholders, desarrolladores, QA, gerentes y usuarios finales que participan en cada etapa del proceso. El equipo humano es el motor del ciclo.
02
📋
Procesos
Conjunto de actividades, metodologías y prácticas definidas que guían la transformación de requisitos en software funcional y de calidad.
03
🛠️
Herramientas
IDEs, sistemas de control de versiones, frameworks de testing, gestores de proyectos y plataformas de despliegue que soportan el desarrollo.
04
📄
Documentación
Artefactos que registran decisiones, requisitos, diseños y resultados de pruebas. Asegura trazabilidad y transferencia de conocimiento.
05
✅
Control de Calidad
Actividades de verificación y validación que aseguran que cada entregable cumple los criterios de aceptación y estándares del proyecto.
06
🔁
Retroalimentación
Mecanismos de revisión y feedback entre fases y con el cliente, que permiten ajustar el rumbo del proyecto antes de que los errores escalen.

Modelo en Cascada

El modelo más clásico y lineal. Cada fase debe completarse al 100% antes de pasar a la siguiente. No hay vuelta atrás (o es muy costosa). Ideal para proyectos con requisitos muy claros y estables desde el inicio.

✓ Ventajas
  • Fácil de entender y gestionar
  • Documentación clara en cada fase
  • Ideal para proyectos pequeños y definidos
  • Progreso medible
✗ Desventajas
  • Poco flexible ante cambios
  • El cliente ve el producto muy tarde
  • Errores tempranos son costosos
  • No apto para proyectos complejos

Modelo Incremental

El desarrollo se divide en pequeños incrementos o módulos funcionales. Cada incremento agrega nuevas funcionalidades al sistema ya existente. El cliente puede usar versiones parciales del sistema mientras se desarrolla el resto.

✓ Ventajas
  • Entregas tempranas de valor
  • Permite retroalimentación continua
  • Riesgos distribuidos
  • Más flexible que Cascada
✗ Desventajas
  • Requiere buena planificación inicial
  • Puede acumular deuda técnica
  • Integración compleja en incrementos grandes
  • Difícil definir el núcleo correcto

Modelo en Espiral

Propuesto por Barry Boehm en 1986. Combina el modelo cascada con la gestión de riesgos. Cada vuelta de la espiral representa una iteración completa que incluye planificación, análisis de riesgos, desarrollo y evaluación.

✓ Ventajas
  • Enfocado en gestión de riesgos
  • Muy flexible y adaptable
  • Ideal para proyectos grandes
  • Retroalimentación continua
✗ Desventajas
  • Complejo de gestionar
  • Costoso en tiempo y recursos
  • Requiere expertos en riesgos
  • No apto para proyectos simples

Metodología Ágil / Scrum

Basado en los 12 principios del Manifiesto Ágil (2001). Divide el proyecto en sprints cortos (1-4 semanas) con entregas funcionales al final de cada iteración. Prioriza personas, colaboración y software funcional sobre documentación.

✓ Ventajas
  • Muy adaptable a cambios
  • Alta colaboración con el cliente
  • Entregas frecuentes de valor
  • Equipos autogestionados
✗ Desventajas
  • Difícil de escalar
  • Requiere compromiso del cliente
  • Documentación limitada
  • Scope creep frecuente

Modelo en V

Extensión del modelo cascada que enfatiza la verificación y validación. Por cada fase de desarrollo (lado izquierdo) hay una fase de prueba correspondiente (lado derecho). La codificación está en el vértice de la V.

✓ Ventajas
  • Énfasis en calidad y testing
  • Pruebas planificadas desde el inicio
  • Estructura clara y ordenada
  • Fácil de supervisar
✗ Desventajas
  • Muy rígido como cascada
  • No apto para requisitos cambiantes
  • No hay prototipos tempranos
  • Costoso ante cambios tardíos
04 — Fases

Fases y sus Objetivos

🔍
1. Planificación & Análisis de Requisitos
Objetivo: Comprender el problema y definir el alcance
Se identifican las necesidades del cliente y se definen los requisitos funcionales y no funcionales del sistema. Se realiza un estudio de viabilidad técnica y económica. Se establece el plan del proyecto con tiempos, costos y recursos. Es la fase más crítica: un error aquí multiplica su costo en etapas posteriores.
Entrevistas Casos de uso SRS Document Factibilidad Cronograma
📐
2. Diseño del Sistema
Objetivo: Definir la arquitectura y estructura del software
Se traduce el "qué" en "cómo". Se diseña la arquitectura del sistema, la base de datos, las interfaces de usuario y los módulos. Se seleccionan tecnologías, patrones de diseño y herramientas. El diseño puede ser de alto nivel (arquitectura) o de bajo nivel (detalle de cada componente).
Diagramas UML ER Diagrams Wireframes Arquitectura Tech Stack
💻
3. Implementación (Codificación)
Objetivo: Construir el software según las especificaciones
Los desarrolladores convierten el diseño en código fuente real. Se siguen estándares de codificación, buenas prácticas y guías de estilo del equipo. Es la fase más visible pero no la más importante: la calidad del código depende directamente de las fases anteriores. Se utilizan sistemas de control de versiones (Git).
Git / GitHub Code Review Unit Testing IDE CI/CD
🧪
4. Pruebas (Testing)
Objetivo: Verificar que el software funciona correctamente
Se verifica que el sistema cumple los requisitos definidos y funciona sin errores. Incluye pruebas unitarias (módulos individuales), de integración (módulos combinados), de sistema (sistema completo) y de aceptación (el usuario valida). Los defectos encontrados se reportan y corrigen antes del despliegue.
Unit Test Integration Test UAT Bug Tracking Performance
🚀
5. Despliegue (Deployment)
Objetivo: Poner el software en producción para los usuarios
El software aprobado se instala en el entorno de producción. Puede hacerse de forma completa (big bang), gradual (rollout), paralela (antigua + nueva) o mediante feature flags. Se realizan capacitaciones a usuarios y se monitorea el sistema. Un despliegue bien planificado minimiza el tiempo de inactividad.
Docker Cloud Deploy Rollback Plan Capacitación Monitoreo
🔧
6. Mantenimiento & Evolución
Objetivo: Mantener el software funcional y actualizado
Una vez en producción, el software requiere mantenimiento continuo. Puede ser correctivo (corregir bugs), adaptativo (ajustar a nuevos entornos), perfectivo (mejorar funcionalidades) o preventivo (evitar futuros problemas). Esta fase es la más larga del ciclo y puede consumir hasta el 60% del costo total del software.
Hotfixes Updates Refactoring Soporte técnico Documentación
